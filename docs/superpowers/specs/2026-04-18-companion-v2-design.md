# Design: Nous Companion v2 — Land a Take, Close the Loop

## Problem Statement

Nous's AI chat companion exhibits two connected failure modes that make it feel like a generic chatbot instead of a thinking partner:

1. **No POV.** AI's replies are rephrases of what the user said, or hedged generic framings ("One hypothesis is..."). It rarely lands its own read of the situation.
2. **No closure sense.** AI keeps asking follow-up questions even after the conversation has arrived somewhere. The user's reply gets shorter, more tired, "ok 啊" — AI still probes for "one more detail." This produces 低营养 (low-nutrition) back-and-forth.

Both symptoms have the same root: **an AI with nothing to say fills silence with questions**. A companion with a real read of the situation lands the read and stops.

## What Makes This Cool

The insight that unlocks the design: **closure is not silence — it's crystallization.** Instead of AI "knowing when to shut up," AI reaches a landing, offers a concrete next step, then proposes to save the conversation as a markdown filed under the user's Project. The markdown is user-owned (exportable to Obsidian), serves as Alex's future-self reference, and gives AI a productive exit ramp that doesn't feel like abandonment.

Every long conversation ends with a durable artifact in the Project. The chat becomes the thinking process; the markdown is the captured insight.

## Constraints

- Must work within the existing `ProvocationJudge` per-turn architecture. Don't rebuild; extend.
- The markdown artifact must be in markdown format (Obsidian-compatible). User's existing workflow ports to external tools.
- Cannot disrupt in-flight work: ChatMode UI removal (PR #11) is the current focus; this design ships after.
- Quick-action openers already hardcode `.companion` — this design tunes only the companion behavior, not strategist.
- Must not regress the "repeat clarification" bug just fixed via `shouldAllowInteractiveClarification`.
- Language: Alex uses Cantonese. Companion prompts stay language-neutral; AI mirrors user language.

## Premises

1. **AI's core value is having a read, not gathering info.** Agree/disagree?
2. **Every companion reply must land a take (POV) — not echo back.** The current "hypothesis framing" in anchor.md softens AI too much. Agree/disagree?
3. **"Good closure" = take + concrete next step + silence, OR take + markdown proposal.** Follow-up questions are the exception, not default. Agree/disagree?
4. **Closure detection belongs in ProvocationJudge (per-turn), not as a separate service.** The judge already reads the whole thread each turn; extending it with `arcStage` is the cheapest architectural fit. Agree/disagree?
5. **Markdown is user-owned artifact stored in the conversation's Project, not a new memory scope.** It does NOT feed back into future AI context automatically — Alex opts in by re-referencing it in a later chat. Agree/disagree?

## Approaches Considered

### Approach A: Prompt-only (minimal viable)

**Summary:** Change `anchor.md` and `ChatMode.companion.contextBlock` to require "land a take, stop if nothing to add, offer next step when relevant." No judge changes, no markdown feature.

**Effort:** S (1-2 hours)
**Risk:** Low — pure prompt edit, easy to revert
**Pros:**
- Ships same-day, immediate behavior signal
- Zero migration risk
- Establishes baseline for measuring whether judge/markdown additions help more
**Cons:**
- No closure mechanism — user still has to end chats the awkward way
- No markdown artifact — conversations still evaporate
- Doesn't solve "wrap-up" — only softens "repetition"
**Reuses:** Existing anchor/context layer assembly in `ChatViewModel.assembleContext`.

### Approach B: Prompt + User-initiated Markdown button

**Summary:** Approach A's prompt changes, plus a "💾 Save as markdown" button in the chat header that drafts + saves a markdown to the current Project. AI never proposes the markdown — purely user-initiated.

**Effort:** M (1 day)
**Risk:** Low-Med — adds a new service and file writes
**Pros:**
- User is always in control (no annoying AI prompts)
- Ships before arc detection is tuned
- Markdown artifact exists; validates the value prop independently
**Cons:**
- Requires user to remember the button exists
- Doesn't give AI a closure signal — still uses follow-ups to fill silence
- Half the design; arc detection is still missing
**Reuses:** `UserMemoryService` pattern for the new markdown service; existing Project model for storage.

### Approach C: Full design — Prompt + Judge arcStage + AI-proposed markdown

**Summary:** Approach B, plus extend `JudgeVerdict` with `arcStage: opening|exploring|insight|landing`. When a turn's verdict lands in `landing` AND the user's recent replies look like acknowledgment (short, affirmative), AI proposes "睇来倾得差唔多，要唔要 save 成 markdown？" inline. User accepts → markdown saves. User declines → AI does not re-prompt in this conversation.

**Effort:** L (2-3 days), but phaseable
**Risk:** Med — arc detection is a new judge signal, needs fixture coverage
**Pros:**
- Closure becomes productive: AI exits with an artifact instead of silence
- Architecturally clean: reuses per-turn judge
- Separates prompt rules (subjective) from closure detection (testable via fixtures)
- AI stops needing follow-ups as a silence-filler; it has a real exit
**Cons:**
- Biggest surface: prompt + judge + service + UI + tests
- Arc detection could misfire — false "landing" reads are annoying
- Obsidian export path must not leak user data unintentionally
**Reuses:** `ProvocationJudge` verdict path, `JudgeVerdict` model, `governanceTelemetry` for judge_events logging, `UserMemoryService` pattern for new `ConversationMarkdownService`.

## Recommended Approach

**Approach C, phased:**

- **Phase 1** (ship first, ~2 hours): Approach A's prompt changes only. Deploy, test manually, observe whether "Land a take, stop" changes response shape noticeably without any other machinery. This answers: is the prompt rewrite sufficient, or does AI need the judge to force its hand?
- **Phase 2** (day 1-2): Add `ConversationMarkdownService` + user-initiated "save as markdown" button (Approach B). Validates markdown's usefulness independently before AI proposes it. Obsidian export confirmed.
- **Phase 3** (day 2-3): Extend `JudgeVerdict` with `arcStage` per the unified schema delta at `docs/superpowers/specs/2026-04-18-judge-verdict-v2-schema.md`. Add arcStage to fixture bank. When landing detected → AI proposes markdown. User-decline sticky within conversation.

**Why phased:** each phase is independently useful. Alex gets value after Phase 1 even if we never ship Phase 3. Phase 2's button is a safety net if arc detection in Phase 3 misfires.

## Open Questions

1. **Prompt language:** Commands stay in English (anchor.md convention) but behavior describes bilingual context ("match Alex's language"). Any risk AI reverts to English when user is Cantonese? Need a fixture scenario.
2. **Arc false positives:** How does AI recover if it proposes markdown too early and user declines? Should decline "stick" for 5 turns, whole conversation, or never re-prompt? Default: whole conversation, reset on new node.
3. **Markdown template:** Initial proposal is `# Title\n\n## Summary\n\n## Key insight\n\n## Next step\n\nSource: [conversation link]`. Is that enough, or does Alex want a "what Alex was feeling" line?
4. **Obsidian export mechanism:** Does Nous write directly to user's Obsidian vault path (requires filesystem picker + bookmark), or copy-to-clipboard as markdown? Cheapest-first = clipboard.
5. **Markdown vs. Entry:** Nous already has `MemoryEntry` for smaller facts. Markdown is bigger, conversation-scoped. Do they coexist forever, or does markdown eventually replace manual entries?

## Success Criteria

- **Qualitative (Alex self-report after 1 week of dogfooding):**
  - "AI feels like it has something to say, not just asking me questions"
  - "When I'm done talking, I can close cleanly — AI offers markdown or shuts up"
  - At least 3 saved markdowns live in Projects, exported to Obsidian, re-read later
- **Quantitative (governance telemetry):**
  - Median conversation length (turns) drops 20-30% — AI not padding with follow-ups
  - Markdown-accept rate > 30% on turns where AI proposes (proposes are valuable, not noise)
  - False-positive markdown proposals (user declines, AI re-proposes in next turn) = 0
